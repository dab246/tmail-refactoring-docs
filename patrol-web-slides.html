Cross-platform E2E testing architecture for Twake Mail —
one test file, all platforms, full SOLID compliance

Abstract Factory Conditional Export SOLID 5 PRs
Context — ADR-0080

Why Patrol on Web?

Twake Mail is multi-platform

  • Android & iOS — already covered by Patrol (ADR-0053)
  • Web — growing user base, no E2E coverage yet
  • Same feature, different widget trees per platform

Patrol 4.x supports Chrome

  • --device=chrome runs Flutter web tests
  • Same CLI, same Docker backend, no new framework
  • Headless mode for CI (--web-headless=true)
  • Web CSS selector access via WebAutomator

Key difference from mobile: Web runs at localhost — no ngrok or jq needed. Same Docker JMAP backend serves both platforms.

Problem — POC on branch patrol/web

3 Structural Problems

SRP Violation

One robot, two platforms:

// Same class
Future<void> addContent() // mobile
Future<void> addContentWeb() // web

Every platform adds methods to existing robots

DRY Violation

Same flow, written twice:

send_email_scenario.dart
web_send_email_scenario.dart

login_with_basic_auth.dart
web_login_with_basic_auth.dart

Duplicate grows with every new test

OCP Violation

Adding a platform = modifying existing files

// Every new platform forces edits
if (platform == 'web') ...
else if (platform == 'desktop') ...
else ...  // mobile

Robots, scenarios, test files all need changes

Decision — ADR-0081

Separate WHAT from HOW

WHAT — Business Flow

Scenario: describes what the user does.
Shared across all platforms.
Written exactly once.

HOW — UI Interaction

Robot: describes how to interact with the UI.
Implemented per platform.
Only the different parts.

Pattern: Abstract Factory + Interface Segregation
Platform selection: Dart conditional exports — no if/else in any logic class

Architecture — ADR-0081

Flow Overview

Test File  (one file per scenario)
    │
    ▼
TestBase.runPatrolTest()
    ├── auto-applies platform tag
    └── calls createRobotFactory($)
              ↑ compile-time conditional export
              ├── MobileRobotFactory  (default)
              └── WebRobotFactory     (dart.library.html)
    │  injects into
    ▼
BaseTestScenario(PatrolIntegrationTester $, RobotFactory robots)
    │  uses
    ▼
Abstract Robot Interfaces  ← scenarios depend only on these
    ├── AbstractComposerRobot
    └── AbstractThreadRobot
         │  implemented by
         ▼
Concrete Robots
    ├── mobile/MobileComposerRobot
    └── web/WebComposerRobot

TestBase is the only class that calls createRobotFactory()

Scenarios and test files never reference any platform-specific class

Platform tag (web/mobile) is auto-applied by TestBase

Robots identical across platforms are reused — no duplication

Key Design #1 — OCP Fix

Dart Conditional Export
— No if/else in Logic

❌ POC approach — OCP violation

// Logic code must change per new platform
if (kIsWeb) return WebRobotFactory($);
return MobileRobotFactory($);

✓ Proposed — idiomatic Dart

// robot_factory_provider.dart — never changes
export 'mobile_robot_factory_provider.dart'
    if (dart.library.html) 'web_robot_factory_provider.dart';

// Each provider exports one function
RobotFactory createRobotFactory(PatrolIntegrationTester $)
    => WebRobotFactory($);
Adding a new platform →

Create a new provider file + factory + robot implementations.
Zero existing files are modified. Same pattern Flutter uses for dart:html vs dart:io.

Key Design #2 — DIP + One Test File

Factory via DI · One File per Scenario

BaseTestScenario — zero platform knowledge

abstract class BaseTestScenario extends BaseScenario {
  final RobotFactory robots;  // injected

  const BaseTestScenario(super.$, this.robots);

  @override
  Future<void> execute() async {
    await robots.loginRobot().login(...);
    await runTestLogic();
  }
}

Scenario — written once, shared

class SendEmailScenario extends BaseTestScenario {
  @override
  Future<void> runTestLogic() async {
    await robots.threadRobot().openComposer();
    await robots.composerRobot().addContent('Hello');
    await robots.composerRobot().send();
  }
}

One test file — all platforms

// tests/composer/send_email_test.dart
void main() {
  TestBase().runPatrolTest(
    description: 'Send email',
    scenarioBuilder: ($, robots)
        => SendEmailScenario($, robots),
  );
}

Folder Layout

integration_test/
├── robots/
│   ├── abstract/          # Interfaces per feature area
│   │   ├── abstract_composer_robot.dart
│   │   └── abstract_thread_robot.dart
│   ├── mobile/            # Mobile implementations
│   │   ├── base_composer_robot.dart  # shared base
│   │   └── mobile_composer_robot.dart
│   └── web/               # Web (only where UI differs)
│       └── web_composer_robot.dart
├── factories/
│   ├── robot_factory.dart            # Abstract interface
│   ├── robot_factory_provider.dart   # Conditional export ← never changes
│   ├── mobile_robot_factory.dart
│   ├── mobile_robot_factory_provider.dart
│   ├── web_robot_factory.dart
│   └── web_robot_factory_provider.dart
├── scenarios/             # All shared — no web_/mobile_ prefix
│   └── composer/send_email_scenario.dart
└── tests/                 # One file per scenario
    └── composer/send_email_test.dart

Abstract layer

One interface per feature area. Scenarios depend only on these — never on concrete robots.

Shared base robot

BaseComposerRobot holds methods identical across platforms (e.g. send()). Mobile and web only override what genuinely differs.

Robot reuse

If a robot is identical on all platforms, both factories return the same class. No web variant needed.

Workflow — ADR-0081

Adding a New Test — 6 Steps

1
UIKeys — Add ValueKey(UiKeys.xxx) to all platform widget variants in lib/features/base/model/ui_keys.dart
2
Abstract robot — Define interface in robots/abstract/ if not yet existing for the feature area
3
Implement — robots/mobile/ and robots/web/. Shared methods → base class. Identical robot → reuse same class in both factories
4
Register — Add the robot to MobileRobotFactory and WebRobotFactory
5
Scenario — Write one scenario in scenarios/ using abstract robots only — no platform awareness
6
Test file — scenarioBuilder: ($, robots) => MyScenario($, robots) — one file, runs on all platforms
SOLID — ADR-0081

SOLID Alignment

PrincipleHow it applies
S — SRP Robot: UI interaction. Scenario: business flow. Factory: object creation. TestBase: lifecycle + wiring.
O — OCP New platform → add provider + factory + robots. No existing class modified.
L — LSP WebRobotFactory and MobileRobotFactory are fully substitutable through RobotFactory.
I — ISP Each robot interface scoped to its feature area. Split RobotFactory if it exceeds ~15 methods.
D — DIP Scenarios depend on RobotFactory (abstraction). TestBase imports createRobotFactory from provider — no concrete factory in logic.

Platform branching exists in exactly one place: the export declaration in robot_factory_provider.dart — which contains zero logic.
